GiveGigs case study

Your schema is the single source of truth for AI tooling.

A manual database change without a migration caused schema drift, making environments, tooling, and AI unreliable. The fix was to restore migration history and sync the schema.

The drift

  • Local environments stopped matching production.
  • New developers couldn't reproduce the database state.
  • Generated types became unreliable.
  • AI-assisted tools started generating code against a schema that didn't exist in source control.

The fix

  1. Identified the untracked schema change.
  2. Removed the incorrect migration state from migration history.
  3. Reapplied the database migrations.
  4. Normalized the schema so source control, migration history, and the database were aligned again.

Lesson

Migration history is part of your application's source code. When database changes bypass the migration process, you create risk for deployments and confuse tooling, type generation, CI pipelines, and AI coding assistants that depend on the schema being reliable.

Never bypass migrationsDon't apply production schema changes without a migration.
Review migrations as codeTreat migration files with the same rigor as application code.
Version-control schema historyStandardize schema changes with permission controls.
Verify environmentsRegularly check that local, staging, and production schemas match.
